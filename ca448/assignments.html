Assignment 1

Assignment 1: A Lexical Analyser for the ImpCA Language

Aim

The aim of this assignment is to implement a lexical analyser using JavaCC for a simple language called ImpCA.

This assignment is worth 10% of the overall marks for this course. It is an individual assignment. Submissions will be checked very closely for signs of plagiarism, for which strict penalties apply.

Lexical Analyser

Case is insignificant in the ImpCA langauge. The lexical analyser should recognise the following tokens:

keywords: and, begin, boolean, const, do, else, end, false, if, integer, main, not, or, program, real, return, string, then, true, var, void, while.

identifiers: Any other string of letters, digits or underscore character ('_') beginning with a letter.

strings: As in C string are delimited by double quotes. Strings may contain any alphanumeric found on a standard keyboard or punctuation characters listed below. Strings may contain quotes, backslashes or newlines only if escaped by a backslash.

numbers: A string of (decimal) digits. Real numbers are represented by a string of digits, a period character "." and a string of digits. Examples of valid numbers are 123, 0.123 and 1.23. Numbers such as 123. and .123 are invalid.

operators, relations and punctuation marks: +  -  *  /  %  =  !=  <  >  <=  >=  (  )  ,  ;  :  .  :=  ?  !

Comments in ImpCA can appear between any two tokens and start with "/-" and end with "-/". Comments can be nested.

Submission

You should submit, by email to David.Sinclair@computing.dcu.ie, all the source files (JavaCC file and other files which were used to generate your parser) in a Winzip file along with a brief description of each of the major components of your lexiscal analyser describing how you implemented them and justifying why you implemented them in the way you did, by 10am on Monday 11th November. The assignment carries 10 marks and late submissions will incur a 1 mark penalty for each 24 hours after the submission.

Assignment 2

Assignment 2: A Top-Down Parser with Semantic Analysis for the ImpCA Language

Aim

The aim of this assignment is to add a parser and semantic analysis checks to the lexical analyser you have implement for the simpL language as Assignment 1. This assignment is worth 15% of the overall marks for this course. It is an individual assignment. Submissions will be checked very closely for signs of plagiarism, for which strict penalties apply. The parser must be implemented using JavaCC.

The syntax analyser should recognise the language described by the following grammar. Tokens are represented the Arial font. Italicised font represents non-terminals. Bold parentheses ( ) are used for grouping. Bold curly braces { } indicate optional items. The vertical bar | represents alternation ("or"). A Kleene star * indicates that the preceding item can be repeated zero or more times and a + indicates that the preceding item can be repeated one or more times. Epsilon ε  denotes the empty string.

program := ( decl )*
                    ( function )*
                    main_prog

decl := ( var_decl | const_decl )*

var_decl := var ident_list:type ( , ident_list:type )* ;

const_decl := const identifier:type = expression ( , identifier:type = expression )* ;
             
function :=
              type identifier ( param_list )
              ( decl )*
              ( statement ; )*
              return ( expression | ε  ) ;

param_list := ( identifier:type ( , identifier:type )* | ε  )

type := integer | boolean | real | string | void

main_prog :=
                   main
                   ( decl )*
                   ( statement ; )*

statement :=
                  identifier := expression
                | identifier := string
                | !expression
                | ?identifier
                | identifier ( arg_list )
                | begin ( statement ; )+ end
                | if condition then statement
                | if condition then statement else statement
                | while ( condition ) do statement
                | ε

expression := fragment ( ( + | - | * | / )  fragment )*

fragment := identifier | number | ( + | - ) fragment | expression

condition :=
                  not expression
                | expression ( = | != | < | > | <= | >=  | and | or) expression
                | identifier

ident_list := identifier ( , identifier )*

arg_list := ( identifier ( , identifier )* | ε  )

The ? operator reads from the standard input and stores the value in a variable, i.e. ?x. The ! operator evaluates an expression and outputs the value to the standard output.

In order to perform the following semantic checks on a ImpCA source file you will have to extend your compiler from Assignment 1 by adding a symbol table and building an abstract syntax tree (AST) for the program you are compiling. Using the symbol table and the AST your compiler should perform the following semantic checks.

  • Is every identifier declared within scope before its is used?
  • Is no identifier declared more than once in the same scope?
  • Is the left-hand side of an assignment a variable of the correct type?
  • Are the arguments of an arithmetic operator the integer or real variables, or integer or real constants?
  • Are the arguments of a boolean operator boolean variables or boolean constants?
  • Is there a function for every invoked identifier?
  • Does every function call have the correct number of arguments?
  • Is every variable both written to and read from?
  • Is every function called?

Feel free to add any additional semantic checks you can think of.

Submission

You should submit, by email to David.Sinclair(at)computing.dcu.ie, all the source files (JavaCC file and other files which were used to generate your parser) in a Winzip file along with a description of your abstract syntax tree structure and symbol table, how you implemented them and how you implemented the semantic checking, by 10am on Monday 16th December. Your description should also include any suggestions you may have to improve the ImpCA language. The assignment carries 20 marks and late submissions will incur a 2 mark penalty for each 24 hours after the submission.
